### Task 1.2: Implement main.rs with CLI argument parsing    [small]

**Description:** Create `src/main.rs` with clap derive macros defining the `Cli` struct and `Commands` enum per ADR-0009. Each subcommand prints a placeholder message and exits with code 0. Wire up error handling with `std::process::exit`.

**Implementation Notes:**
```rust
use clap::{Parser, Subcommand};
use std::path::PathBuf;

#[derive(Parser)]
#[command(name = "skilleton", about = "Build and validate agent skills")]
struct Cli {
    #[command(subcommand)]
    command: Commands,
}

#[derive(Subcommand)]
enum Commands {
    /// Initialize a new skill directory layout
    Init { path: PathBuf },
    /// Validate a skill's structure and references
    Check { path: PathBuf },
    /// Build a skill into Markdown output
    Build { path: PathBuf },
}
```

**Test & Acceptance Criteria:**
- [x] `cargo build` produces the `skilleton` binary
- [x] `skilleton --help` displays help text with all three subcommands
- [x] `skilleton init --help`, `skilleton check --help`, `skilleton build --help` each display subcommand-specific help
- [x] Unknown subcommands produce a usage error (exit code 2)

**Dependencies:** Task 1.1

## Stage 2: Rendering Pipeline

### Task 2.1: Implement render module with policy-first ordering    [medium]

**Description:** Create `src/render.rs` with a public `render_skill(skill: &Skill) -> String` function. The function walks the Skill tree depth-first, emitting Markdown with policy-first ordering at every hierarchy level per ADR-0010's rendering spec. Include colocated unit tests per AGENTS.md P-19.

**Rendering rules (from ADR-0010 Decision):**

1. **Skill level:** `#` title → description → `## Policies` → `## Criteria` → `## Procedures`
2. **Procedure level:** `###` heading → policies → criteria → entrance criteria → steps → exit criteria
3. **Step level:** `####` heading → policies → criteria → tasks → completion criteria
4. **Task level:** list item with code-formatted ID, bold subject, action text, optional invokes annotation
5. **Type-specific renderers:**
   - `render_policy` → blockquote `> **{id}**: {text}`
   - `render_criterion` → list item `- **{id}**: {description}`
   - `render_criterion_ref` → list item `- {id}`
   - `render_conditions` → italic annotation `*Conditions: ref1, ref2, ...*` when non-empty
6. **Empty sections** are omitted entirely
7. **Intentional omissions:** `Policy.compatible_with` (validator metadata, not rendered)

**Implementation Notes:**
- Add `pub mod render;` to `src/lib.rs`
- Use `String` concatenation or `std::fmt::Write` for building output
- Each hierarchy level is a helper function: `render_procedure`, `render_step`, `render_task`, `render_policy`, `render_criterion`, `render_criterion_ref`, `render_conditions`

**Test & Acceptance Criteria:**
- [x] `render_skill` produces Markdown with `## Policies` before `## Procedures` at skill level
- [x] Procedure-level output has policies before steps
- [x] Step-level output has policies before tasks
- [x] `render_criterion` emits ID and description; `render_criterion_ref` emits ID only
- [x] `render_conditions` emits italic annotation for non-empty conditions, nothing for empty
- [x] Empty sections (no policies, no criteria) are omitted — no empty headings
- [x] Task rendering includes task ID, subject, action, and optional invokes
- [x] Policy rendering uses blockquote format
- [x] Items within a section are emitted in definition order (Vec ordering preserved)
- [x] All unit tests colocated in `src/render.rs` per P-19
- [x] `cargo test` passes with all render tests

**Dependencies:** None (uses existing types from `src/types/`)

**ADR Reference:** ADR-0010, Decision §Rendering module, §Rendering rules by type

## Stage 3: CLI Commands

### Task 3.1: Implement init command    [small]

**Description:** Implement the `init` subcommand in `src/main.rs`. Constructs a default `Skill` struct with placeholder metadata (name derived from directory name, empty description, generated `skill:<name>` ItemId, empty policies/criteria/procedures) and writes it to disk using `SkillWriter::write`. Creates the `<path>/procedures/` directory.

**Implementation Notes:**
```rust
// Derive skill name from the directory name
let name = path.file_name().unwrap_or("my-skill").to_string_lossy();
let skill = Skill {
    meta: SkillMeta { name: name.to_string(), description: String::new() },
    metadata: ItemMeta {
        id: ItemId::parse(&format!("skill:{name}")).unwrap(),
        conditions: vec![],
    },
    procedures: vec![],
    policies: vec![],
    criteria: vec![],
};
SkillWriter::write(&path, &skill)?;
```

**Test & Acceptance Criteria:**
- [x] `skilleton init <path>` creates `<path>/skill.toml`
- [x] `skilleton init <path>` creates `<path>/procedures/` directory
- [x] The generated directory is loadable by `SkillLoader::load` (round-trip validation)
- [x] `skill.toml` contains the correct name derived from the directory name
- [x] Init on an existing directory reports an error and exits with code 1
- [x] Exit code 0 on success

**Dependencies:** Task 1.2

**ADR Reference:** ADR-0009, Decision §Init behavior

### Task 3.2: Implement check command    [medium]

**Description:** Implement the `check` subcommand. Loads a skill via `SkillLoader::load`, then runs all four validators. Collects all findings and reports them to stderr. Exits with code 0 if all pass, code 1 if any fail.

**Implementation Notes:**
Validators to run in order:
1. `validate_invocation_references(&skill)`
2. `validate_criterion_references(&skill)`
3. `validate_type_prefixes(&skill)`
4. `detect_policy_overlaps(&skill)`

Report all errors — do not short-circuit on the first failure.

**Test & Acceptance Criteria:**
- [x] `skilleton check <valid-skill>` exits with code 0
- [x] `skilleton check <invalid-skill>` exits with code 1 and reports errors to stderr
- [x] All four validators are invoked (invocation references, criterion references, type prefixes, policy overlaps)
- [x] Multiple validation errors are all reported, not just the first
- [x] Missing or unreadable path produces a clear error message and exit code 1

**Dependencies:** Task 1.2

**ADR Reference:** ADR-0009, Decision §Check behavior

### Task 3.3: Implement build command    [small]

**Description:** Implement the `build` subcommand. Loads a skill, validates it (same as check), and renders Markdown to stdout using `render_skill`. If validation fails, reports errors to stderr and exits with code 1 without producing output.

**Test & Acceptance Criteria:**
- [x] `skilleton build <valid-skill>` outputs Markdown to stdout
- [x] Output starts with `# <skill-name>` heading
- [x] Output has policies before procedures
- [x] `skilleton build <invalid-skill>` exits with code 1 with errors on stderr, no stdout output
- [x] Build output is self-contained readable Markdown (standalone skill document)
- [x] Exit code 0 on success

**Dependencies:** Task 1.2, Task 2.1

**ADR Reference:** ADR-0009, Decision §Build behavior; ADR-0010, Decision
