### Task 1.2: Implement custom serde for ItemId, CriterionRef, and base types    [medium]

**Description:** Implement custom `Serialize`/`Deserialize` for `ItemId` so it serializes as its string representation (the hierarchical path), not as a struct. Implement serde for `CriterionRef` which wraps `ItemId`. Derive `Serialize`/`Deserialize` on `ItemMeta` and `SkillMeta`. Unit tests per P-19 (colocated in each source file).

**Implementation Notes:**

`ItemId` custom serde — serialize as the string representation:
```rust
impl Serialize for ItemId {
    fn serialize<S: Serializer>(&self, s: S) -> Result<S::Ok, S::Error> {
        s.serialize_str(self.as_str())
    }
}

impl<'de> Deserialize<'de> for ItemId {
    fn deserialize<D: Deserializer<'de>>(d: D) -> Result<Self, D::Error> {
        let s = String::deserialize(d)?;
        ItemId::parse(&s).map_err(serde::de::Error::custom)
    }
}
```

`CriterionRef` — serialize through inner `ItemId`:
```rust
impl Serialize for CriterionRef {
    fn serialize<S: Serializer>(&self, s: S) -> Result<S::Ok, S::Error> {
        self.0.serialize(s)
    }
}

impl<'de> Deserialize<'de> for CriterionRef {
    fn deserialize<D: Deserializer<'de>>(d: D) -> Result<Self, D::Error> {
        ItemId::deserialize(d).map(CriterionRef)
    }
}
```

`ItemMeta` and `SkillMeta` — derive serde:
```rust
#[derive(Debug, Clone, PartialEq, Serialize, Deserialize)]
pub struct ItemMeta { ... }

#[derive(Debug, Clone, Default, PartialEq, Serialize, Deserialize)]
pub struct SkillMeta { ... }
```

### Task 2.1: Effective policy set computation    [medium]

**Description:** Create a `conflict` module (`src/conflict.rs`). Implement a function that walks the skill type hierarchy and computes the effective policy set at each node. A node's effective set is its own policies plus all ancestor policies. For cross-procedure invocations (ADR-0004), merge the caller's effective policies with the callee's effective policies at the invocation point. Export the module from `lib.rs`.

**Implementation Notes:**

The core function signature:
```rust
/// Represents a node in the hierarchy with its effective policies.
pub struct EffectivePolicies {
    /// The ItemId of this node (Skill, Procedure, or Step).
    pub scope: ItemId,
    /// Policies that apply at this scope (own + inherited).
    pub policies: Vec<PolicyOrigin>,
}

/// Tracks where a policy came from.
pub struct PolicyOrigin {
    pub policy: Policy,
    /// The scope where this policy was defined.
    pub origin: ItemId,

pub fn compute_effective_policies(skill: &Skill) -> Vec<EffectivePolicies> { ... }
```

Walk the hierarchy: Skill → Procedure → Step. At each level, accumulate policies from ancestors. For cross-procedure invocations, look up the callee Procedure's effective policies and merge with the caller's scope.

**Description:** Implement overlap detection that identifies cross-origin policy convergence. Report overlaps only when policies from different origins converge: (a) cross-procedure invocations where caller and callee effective sets merge, and (b) multiple policies defined at the same hierarchy level. Single-branch inheritance (child inheriting parent's policies along one ancestry chain) is normal layering and is NOT reported. Return a structured `Vec<PolicyOverlap>` report.

**Implementation Notes:**

### Task 2.3: Compatibility annotations and overlap suppression    [medium]

**Description:** Implement the annotation mechanism for suppressing known-good policy overlaps per ADR-0007 Decision §4. The `compatible_with: Vec<ItemId>` field added to `Policy` in Task 1.3 provides the data model. This task implements the suppression logic in `detect_policy_overlaps`: when two policies at a convergence point have mutual `compatible_with` annotations listing each other, the overlap is suppressed from the report. Constraints: (a) annotation targets a specific overlap (names the other policy), (b) co-located with the policy definition (field on Policy struct), (c) per-overlap — no global rules.

**Implementation Notes:**

```rust
/// Check if two policies have mutual compatibility annotations.
fn are_compatible(a: &Policy, b: &Policy) -> bool {
    let a_id = &a.meta.id;
    let b_id = &b.meta.id;
    a.compatible_with.contains(b_id) && b.compatible_with.contains(a_id)
}
```

In `detect_policy_overlaps`, after computing overlaps, filter out any overlap where ALL pairs of converging policies have mutual `compatible_with` annotations.

**Test & Acceptance Criteria:**
- [ ] Mutual `compatible_with` annotations suppress the corresponding overlap
- [ ] Unilateral annotation (only one policy lists the other) does NOT suppress the overlap
- [ ] Empty `compatible_with` (default) has no effect on overlap detection
- [ ] Annotations do not affect overlaps involving other, non-annotated policies at the same scope
- [ ] Suppressed overlaps are excluded from the returned `Vec<PolicyOverlap>`
- [ ] All unit tests are colocated per P-19

**Dependencies:** Tasks 2.1, 2.2

**ADR Reference:** ADR-0007, Decision §4 (compatibility annotations with three design constraints)

**Implementation Notes:**

```rust
/// Storage interface for skill definitions.
pub trait SkillRepository {
    /// Load a skill by name (the slug from its ItemId).
    fn load_skill(&self, name: &str) -> Result<Skill, RepositoryError>;
    /// List available skill names.
    fn list_skills(&self) -> Result<Vec<String>, RepositoryError>;
    /// Find an item by its ItemId within a named skill.
    fn find_item(&self, skill: &str, id: &ItemId) -> Result<Option<ItemRef>, RepositoryError>;

/// Owned reference to a specific item kind.
pub enum ItemRef {
    Skill(Skill),
    Procedure(Procedure),
    Step(Step),
    Task(Task),
    Policy(Policy),
    Criterion(Criterion),
}

/// Errors from repository operations.
pub enum RepositoryError {
    /// Skill not found by name.
    NotFound(String),
    /// Storage-layer error (I/O, parse, etc.).

}
```
