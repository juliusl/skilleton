### Test-Gap Findings

**TG1.1 — Existing test `multiple_policies_at_same_level_produce_overlap` will break.** The current test at `conflict.rs:442` asserts that 2 policies at the same level produce an overlap. Task 1.3 changes the threshold to `> 2`. This test must be updated to use 3 policies, but the plan's acceptance criteria do not mention updating this existing test — they only say "add test" for three policies. The implementer could add the new test while leaving the old one broken, violating P-2.

**TG1.2 — No backward-compatibility test for existing TOML files with explicit empty arrays.** Task 1.1 adds `#[serde(default)]` to Vec fields. The acceptance criteria test that omitted fields deserialize to empty Vecs, but do not test that existing TOML files with explicit empty arrays (`tasks = []`, `policies = []`) still deserialize correctly. If `serde(default)` interacts unexpectedly with explicit empty arrays, existing on-disk files could fail to load.

**TG1.3 — `write_to()` missing stale procedure cleanup logic.** The existing `write()` method (writer.rs:28-46) removes stale `.toml` files from the procedures directory before writing. The plan's `write_to()` implementation notes say "same write logic as `write()`" but do not explicitly mention this cleanup step. If omitted, renamed or deleted procedures persist on disk after re-init.

**TG1.4 — Task 1.3 threshold change has an off-by-one edge case at exactly 3 policies where a compatible pair exists.** The acceptance criteria test "three incompatible policies" but not "three policies where two are compatible and one is not." The `all_pairs_compatible` check handles this correctly (returns false if any pair is incompatible), but the plan has no acceptance criterion that exercises this mixed case.

**TG1.5 — Onboarding fixture has 2 policies at skill level but also 1 policy at procedure level and 1 at step level.** Task 1.3 acceptance criterion "onboarding fixture produces zero overlap warnings" will pass because all scopes have ≤2 policies. But the test is not sensitive to the threshold logic — it would also pass with the threshold at `> 0` or `> 100`. The test proves absence of warnings but does not prove correctness of the heuristic.

## Stage 2: Template Rendering Engine

### Security Checks

- [ ] **S2.1** — No user-supplied strings interpolated into SQL, shell commands, or file paths — **Finding:** Task 2.3 reads a user-supplied `--template <path>`. The path is passed to `std::fs::read_to_string()` which follows symlinks. A symlink to `/etc/passwd` would read that file's contents and use it as a template (producing garbled output, not a data leak, since the template is applied to a RenderContext and written to stdout). Low risk but noted.
- [x] **S2.2** — No secrets, credentials, or API keys in committed files
- [ ] **S2.3** — No deserialization of untrusted input without validation — **Finding:** `ramhorns` parses user-supplied template files. Mustache is logic-less so code execution risk is minimal, but malformed templates could cause panics in `ramhorns` if error handling uses `expect()` instead of `?`. The plan's implementation notes show `expect("default template must parse")` — this is acceptable for the compiled-in default, but user-supplied templates must use fallible parsing with proper error propagation.
- [ ] **S2.4** — Dependencies pinned to specific versions (no wildcards) — **Finding:** The plan does not specify the `ramhorns` version. The implementer must pin it (e.g., `ramhorns = "0.14"` not `ramhorns = "*"`).
- [x] **S2.5** — File permissions on created artifacts are not overly permissive — output goes to stdout, no files created
- [ ] **S2.6** — Any new external input surface has validation at the boundary — **Finding:** The `--template` flag introduces a new input surface (file path + file contents). Path validation (exists, readable) is specified in the plan. Template syntax validation (valid Mustache) is specified. But there is no size limit on the template file — a multi-gigabyte file would be read entirely into memory.

### UX Checks

- [x] **U2.1** — Every error path produces a human-readable message on stderr — plan specifies "clear error to stderr with context" for template errors
- [x] **U2.2** — Every user-facing command exits with code 0 on success, non-zero on failure — plan specifies exit code 1 for template errors
- [x] **U2.3** — Invalid input is rejected with a helpful message, not a crash — plan specifies clear error for missing file and invalid syntax
- [x] **U2.4** — Resources cleaned up on error paths — template rendering is pure string manipulation, no resources to leak
- [x] **U2.5** — If the stage writes data, there is a way to read it back — output is to stdout, verifiable by pipe/redirect
- [x] **U2.6** — If the stage creates state, there is a way to inspect the new state — no persistent state created
- [x] **U2.7** — A user who did not write the code can verify the stage's output — Markdown output is human-readable; diff against previous output verifies backwards compatibility

### Test-Gap Findings

**TG2.1 — Mustache HTML-escaping will break byte-for-byte backwards compatibility.** The Mustache spec requires `{{variable}}` to HTML-escape `&`, `<`, `>`, `"`. The `ramhorns` crate follows this spec. If any policy text, description, criterion text, or action contains `&`, `<`, or `>`, the template output will differ from the current `write!`/`writeln!` output (which does not escape). The plan's acceptance criteria say "byte-for-byte backwards compatibility" but do not account for this. The fix is to use triple-mustache `{{{variable}}}` (unescaped) in the default template, but the plan's template example in ADR-0011 uses double-mustache `{{variable}}`.

**TG2.2 — Template whitespace matching is fragile.** The current `render_skill()` uses precise `writeln!(out)` calls to produce blank lines between sections. Mustache template engines often produce extra or missing blank lines due to standalone tag handling (tags on their own line are removed along with the line, but this interacts with newlines before/after sections). Achieving byte-for-byte match requires careful template authoring and testing with multiple fixture inputs, not just the onboarding fixture.

**TG2.3 — `expect()` on default template parse is a hidden panic.** The plan shows `Template::new(template).expect("default template must parse")` in `render_skill_with_template()`. This function accepts any `&str` template — including user-supplied ones from `--template`. If the implementer uses `expect()` for all template parsing (not just the default), invalid user templates will panic instead of returning an error. The plan's Task 2.3 specifies error handling for invalid syntax, but the implementation pattern in Task 2.2 shows `expect()`.

**TG2.4 — No test for RenderContext with special characters.** The acceptance criteria test `RenderContext` from an empty skill and from the onboarding fixture. The onboarding fixture uses plain ASCII text. No test exercises Mustache-significant characters (`{`, `}`, `&`, `<`, `>`) in policy text, descriptions, or names. These characters will be escaped or misinterpreted by the template engine.

**TG2.5 — No test for `render_skill_with_template()` with a template that references nonexistent context fields.** Mustache silently renders missing variables as empty strings. A user template with `{{nonexistent_field}}` will produce empty output for that variable with no warning. The plan has no acceptance criterion for this behavior and no guidance on whether to warn users.

**TG2.6 — Task task format uses backtick-wrapped ID.** The current `render_task` output is `` - `task:t1` **Subject**: Action ``. The backtick wrapping requires special handling in the template — the template must include literal backticks around `{{id}}`. This is easy to get wrong and the plan's template example in ADR-0011 does not show backtick wrapping for task IDs.

**TG2.7 — No backward-compatibility test for empty-description skill.** The current `render_skill()` omits the description line when `description.is_empty()`. The `RenderContext` must handle this with `has_description: bool`. The plan's acceptance criteria reference "empty skill has all `has_*` fields false" but don't verify that the rendered output for an empty-description skill matches the current hardcoded output (no extra blank line).
