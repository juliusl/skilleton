### Test-Gap Findings

| ID | Gap | Severity | Detail |
|----|-----|----------|--------|
| TG1.1 | No boundary test for exactly 50-char slug | Medium | Tests cover >50 (rejected) but not exactly 50 (should be accepted). Off-by-one in the `>` vs `>=` check would go undetected. |
| TG1.2 | No test for empty slug after colon | Medium | Input like `"skill:"` produces an empty slug. `validate_slug` rejects empty slugs, but no test exercises this path through `parse`. |
| TG1.3 | No test for slugs containing digits | Low | `validate_slug` allows `ascii_digit`, but no test confirms `"skill:auth2"` parses successfully. |
| TG1.4 | No test for multiple colons in a segment | Low | Input like `"skill:foo:bar"` — `split_once(':')` takes the first colon, leaving `"foo:bar"` as the slug. The colon would fail `validate_slug`'s charset check, but this is untested. |
| TG1.5 | No test for `append()` with invalid slug | Low | `append()` calls `validate_slug`, but no test confirms it returns `ItemIdError` for bad input. |
| TG1.6 | No test for `segments()` on a single-segment path | Low | `segments_returns_correct_parts` tests a 2-segment path. No test confirms the 1-segment case returns exactly one segment. |
| TG1.7 | Hierarchy ordering not enforced | Info | `ItemId::parse` accepts `"step:x.skill:y"` (child-type before parent-type). ADR-0003 specifies the canonical path runs root-to-leaf, but parsing does not enforce this. This may be intentional — the plan does not mention hierarchy-order validation. Flag for awareness. |

### UX Checks

| ID | Check | Result | Notes |
|----|-------|--------|-------|
| U2.1 | Error messages on stderr | N/A | Returns `Result<(), Vec<ReferenceError>>`. No stderr. |
| U2.2 | Exit codes | N/A | Library crate. |

### Test-Gap Findings

| ID | Gap | Severity | Detail |
|----|-----|----------|--------|
| TG2.1 | `detect_cycle` returns only the first cycle | Medium | If a Skill has two independent cycles (e.g., A→B→A and C→D→C), only one `CycleDetected` error is returned. The plan's acceptance criteria say "cycle detected" (singular), but users may need all cycles reported. |
| TG2.2 | No test for diamond DAG (valid) | Medium | A diamond graph (A→B, A→C, B→D, C→D) is a valid DAG with shared descendants. No test confirms it passes. This is a common edge case for DFS-based cycle detection. |
| TG2.3 | No test for empty Skill (zero procedures) | Low | `validate_references` on a Skill with `procedures: vec![]` should return `Ok(())`. Untested. |
| TG2.4 | No test for multiple `invokes` from same procedure | Low | A procedure with two tasks each invoking different procedures — tests only have one task per step per procedure. |
| TG2.5 | No test that verifies cycle path contents | Medium | `direct_cycle_detected` and `indirect_cycle_detected` check `matches!(e, CycleDetected { .. })` but do not assert the `cycle` vec contains the correct node sequence. A buggy cycle extraction could pass these tests. |
| TG2.6 | No stack-depth protection for deep graphs | Info | `detect_cycle` uses recursive DFS. A Skill with thousands of chained procedures could stack-overflow. Unlikely in practice (Skills are small), but worth noting for future-proofing. Flag for awareness. |
| TG2.7 | HashMap iteration order is nondeterministic | Info | Which cycle is found first depends on HashMap key iteration order. This is correct behavior (any cycle is a valid error) but means test assertions on specific cycle paths may be flaky if tightened. |

### Won't Fix — rationale

- **#4 — No test for `append()` error path:** Error path is exercised indirectly via slug validation tests; dedicated test is low priority.
- **#8 — No test for multiple invokes from same procedure:** Existing DAG tests exercise multiple edges; dedicated test is low priority.
- **#9 — No empty-Skill test:** Empty Skill is trivially correct (no procedures = no references); low value test.
- **#10 — Hierarchy ordering not enforced:** Hierarchy ordering is intentionally not enforced at parse time — the canonical ordering is a convention, not a constraint.
