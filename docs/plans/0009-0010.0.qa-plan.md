## Stage 1: Foundation

### Security Checks

- [ ] **S1.1** вЂ” `clap` version in `Cargo.toml` is pinned to `"4"` (major only) вЂ” verify no `*` or `>=` wildcards; confirm `Cargo.lock` pins an exact version after `cargo check`
- [ ] **S1.2** вЂ” `src/main.rs` stub contains no hardcoded secrets, tokens, or API keys
- [ ] **S1.3** вЂ” No deserialization in this stage (clap derive macros parse CLI args, not untrusted data) вЂ” confirm no `serde` usage in `main.rs`
- [ ] **S1.4** вЂ” `clap = { version = "4", features = ["derive"] }` is the only new dependency; `Cargo.lock` captures the resolved version
- [ ] **S1.5** вЂ” No files are created on disk by this stage вЂ” N/A, confirm no `fs::write` or `fs::create_dir` calls in the stub `main.rs`
- [ ] **S1.6** вЂ” `PathBuf` arguments from clap are raw user input вЂ” verify they are not used in this stage beyond parsing (actual file operations deferred to Stage 3)

### UX Checks

- [ ] **U1.1** вЂ” `skilleton` invoked with no subcommand produces a usage error on stderr (clap default behavior) вЂ” verify message is readable
- [ ] **U1.2** вЂ” `skilleton --help` exits with code 0; unknown subcommands exit with code 2
- [ ] **U1.3** вЂ” `skilleton init` without a `<path>` argument produces a clap error message naming the missing argument, not a panic or stack trace
- [ ] **U1.4** вЂ” N/A вЂ” no resources opened in this stage
- [ ] **U1.5** вЂ” N/A вЂ” no data written in this stage
- [ ] **U1.6** вЂ” `skilleton --version` (if wired) or `skilleton --help` allows inspecting the binary's identity
- [ ] **U1.7** вЂ” Running `skilleton --help` and visually confirming all three subcommands (`init`, `check`, `build`) are listed with descriptions

### Test-Gap Findings

- **TG1.1** вЂ” The plan tests `exit code 2` for unknown subcommands but does not test missing required arguments (e.g., `skilleton init` with no path). clap produces exit code 2 for both, but the error messages differ вЂ” worth verifying.
- **TG1.2** вЂ” No test for `--version` flag behavior. If `#[command(version)]` is added to the `Cli` struct, it should be verified; if omitted, confirm it is intentional.

## Stage 2: Rendering Pipeline

### Security Checks

- [ ] **S2.1** вЂ” `render_skill` uses `&Skill` (in-memory struct) as input вЂ” verify no file path interpolation or shell command construction in `src/render.rs`
- [ ] **S2.2** вЂ” No secrets or credentials in test fixtures embedded in `src/render.rs` colocated tests
- [ ] **S2.3** вЂ” `render_skill` does not deserialize input вЂ” it receives a pre-parsed `&Skill` reference. Confirm no `toml::from_str` or `serde_json` calls in the render module
- [ ] **S2.4** вЂ” No new dependencies added in this stage вЂ” `src/render.rs` uses only `std::fmt::Write` or `String` concatenation
- [ ] **S2.5** вЂ” No files created on disk by the render module вЂ” output is a `String` return value
- [ ] **S2.6** вЂ” Skill field values (names, descriptions, policy text) are emitted verbatim into Markdown вЂ” verify that no user-supplied text could break Markdown structure in a way that changes rendered semantics (e.g., `# ` in a policy text producing a false heading)

### UX Checks

- [ ] **U2.1** вЂ” `render_skill` is a pure function returning `String` вЂ” error paths (if any) should use `Result` or `Option`, not `panic!`/`unwrap`
- [ ] **U2.2** вЂ” N/A вЂ” no CLI command in this stage (render is a library function)
- [ ] **U2.3** вЂ” If `render_skill` receives a `Skill` with empty `name` or `description`, it should produce valid Markdown (empty heading `# ` or omitted), not crash
- [ ] **U2.4** вЂ” No resources to clean up вЂ” pure function
- [ ] **U2.5** вЂ” The render output is a `String` вЂ” the caller (`build` command in Stage 3) prints it to stdout, making it readable
- [ ] **U2.6** вЂ” N/A вЂ” no persistent state created
- [ ] **U2.7** вЂ” Render output is Markdown readable by any human with a text editor вЂ” verify output against ADR-0010's spec by visual inspection of test output

### Test-Gap Findings

- **TG2.1** вЂ” The plan tests section ordering (policies before procedures) but does not test **heading levels** explicitly. Verify `#` for skill title, `##` for Policies/Criteria/Procedures, `###` for procedures, `####` for steps.
- **TG2.2** вЂ” No test for skills with `conditions` on `ItemMeta`. The `render_conditions` function should be exercised with both empty and non-empty conditions vectors.
- **TG2.3** вЂ” No test for Markdown injection via user-supplied text. If a policy's `text` field contains `> ` or `# `, the rendered output could be structurally ambiguous. Not a security risk (output is local), but affects readability.
- **TG2.4** вЂ” No test for a skill with zero procedures. `render_skill` on a policies-only skill should omit the `## Procedures` heading entirely per the empty-section rule.
- **TG2.5** вЂ” The plan says "Items within a section are emitted in definition order (Vec ordering preserved)" but provides no explicit test asserting order when multiple policies or criteria exist. A test with 3+ policies verifying output order would close this gap.

## Stage 3: CLI Commands

### Security Checks

- [ ] **S3.1** вЂ” `init` command uses `path` from CLI args to create directories вЂ” verify `SkillWriter::write` does not interpolate the path into shell commands; confirm it uses `std::fs` APIs only
- [ ] **S3.2** вЂ” No secrets in the default `Skill` struct generated by `init` (placeholder name, empty description, generated ItemId)
- [ ] **S3.3** вЂ” `check` and `build` commands load skill files via `SkillLoader::load`, which deserializes TOML вЂ” verify `SkillLoader` validates the TOML schema (pre-existing; confirm no regression)
- [ ] **S3.4** вЂ” No new dependencies added in Stage 3 вЂ” all logic uses existing library crate types
- [ ] **S3.5** вЂ” `init` creates `skill.toml` and `procedures/` directory вЂ” verify file permissions are inherited from the parent directory (default `std::fs::create_dir_all` behavior), not set to 0o777
- [ ] **S3.6** вЂ” The `path` argument from the user is used as a filesystem path вЂ” verify path traversal is bounded (e.g., `../../../etc/` is handled by the OS, but `init` should not follow symlinks into unexpected locations; `SkillWriter::write` creates a subdirectory under the given path, limiting scope)

### UX Checks

- [ ] **U3.1** вЂ” `init` on an existing directory prints a human-readable error to stderr (e.g., "Error: skill already exists at <path>") вЂ” not a raw `IoError` debug dump
- [ ] **U3.2** вЂ” `init` exits 0 on success, `check` exits 0 on valid skill / 1 on invalid, `build` exits 0 on success / 1 on validation failure
- [ ] **U3.3** вЂ” `check <nonexistent-path>` produces "Error: path not found" or similar, not a panic; `build <nonexistent-path>` same behavior
- [ ] **U3.4** вЂ” If `init` partially creates files and then fails (e.g., permission denied on `procedures/`), the partial state is either cleaned up or clearly reported вЂ” verify no silent partial writes
- [ ] **U3.5** вЂ” `init` writes `skill.toml` вЂ” loadable by `SkillLoader::load` (round-trip). `build` outputs Markdown to stdout вЂ” pipeable to a file
- [ ] **U3.6** вЂ” After `init`, running `check` on the same directory exits 0 вЂ” the initialized state is inspectable and valid
- [ ] **U3.7** вЂ” A new user can run `skilleton init my-skill && skilleton check my-skill && skilleton build my-skill` and understand the output at each step without reading source code

### Test-Gap Findings

- **TG3.1** вЂ” The plan does not test `build` with a skill that has validation errors **and** is otherwise renderable. The spec says "no stdout output" on validation failure вЂ” verify stdout is empty (not just that stderr has errors).
- **TG3.2** вЂ” No test for `init` when the parent directory does not exist (e.g., `skilleton init /nonexistent/parent/my-skill`). Should this create parent directories or fail?
- **TG3.3** вЂ” `check` acceptance criteria says "all four validators are invoked" but does not specify how to verify this. Consider a test with a skill that triggers each validator individually to confirm coverage.
- **TG3.4** вЂ” No test for `build` output being valid, well-formed Markdown. The plan says "self-contained readable Markdown" but has no structural assertion beyond "starts with `# <skill-name>`" and "policies before procedures."
- **TG3.5** вЂ” The `init` command derives the skill name from `path.file_name()`. No test for edge cases: path ends with `/`, path is `.`, path is `..`, path contains special characters.

## Stage 4: Validation

### Security Checks

- [ ] **S4.1** вЂ” Toy fixture at `tests/fixtures/onboarding/` contains no user-supplied strings at runtime вЂ” it is static test data. Verify no placeholder credentials in fixture TOML files
- [ ] **S4.2** вЂ” No secrets in fixture files or integration test source (`tests/cli.rs`)
- [ ] **S4.3** вЂ” Integration tests invoke `skilleton` binary via `std::process::Command` вЂ” verify test assertions do not eval stdout content as code
- [ ] **S4.4** вЂ” If `assert_cmd` or `tempfile` is added as a dev-dependency, pin to a specific version range
- [ ] **S4.5** вЂ” Integration tests using temp directories (via `tempfile` crate) should clean up on completion вЂ” `tempfile::tempdir()` auto-cleans on drop
- [ ] **S4.6** вЂ” Integration tests pass untrusted-like input to the CLI (e.g., invalid paths) вЂ” verify these tests exercise the validation boundary, not bypass it

### UX Checks

- [ ] **U4.1** вЂ” Integration tests for `check <invalid-dir>` verify that error output on stderr is human-readable, not a debug struct dump
- [ ] **U4.2** вЂ” All integration tests assert specific exit codes (0 or 1), not just "did not panic"
- [ ] **U4.3** вЂ” Integration test for `skilleton check <nonexistent>` verifies the error message mentions the path that was not found
- [ ] **U4.4** вЂ” Temp directories created by integration tests are cleaned up automatically via `Drop` вЂ” no leaked directories after `cargo test`
- [ ] **U4.5** вЂ” The toy fixture at `tests/fixtures/onboarding/` is loadable by `SkillLoader::load` вЂ” the fixture is its own verification artifact
- [ ] **U4.6** вЂ” The fixture's structure (policies, criteria, procedures, steps, tasks) is inspectable by running `skilleton check tests/fixtures/onboarding/` manually
- [ ] **U4.7** вЂ” Documentation (Task 4.3) includes copy-pasteable commands that a new developer can run to verify the CLI works

### Test-Gap Findings

- **TG4.1** вЂ” The plan creates one toy fixture (`onboarding`). There is no negative fixture вЂ” a skill with known validation errors вЂ” to test `check` failure paths in integration. The plan mentions `<invalid-dir>` but this likely means a non-skill directory, not a structurally invalid skill.
- **TG4.2** вЂ” Integration tests exercise the binary via `std::process::Command`, which tests the compiled binary. If the binary name changes or the build profile differs, tests may silently pass against a stale binary. Consider using `cargo_bin` or `assert_cmd` to locate the correct binary.
- **TG4.3** вЂ” No integration test for `build` output piped to a file and then diffed against an expected snapshot. Without snapshot testing, Markdown format regressions are invisible to CI.
- **TG4.4** вЂ” Documentation (Task 4.3) has acceptance criteria for "examples show expected exit codes" but no criteria for accuracy вЂ” the examples could show wrong exit codes and still satisfy the checkbox.
