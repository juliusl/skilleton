## Stage 2: Rendering Pipeline

### Security Checks

- [x] **S2.1** ‚Äî `render_skill` uses `&Skill` (in-memory struct) as input ‚Äî verify no file path interpolation or shell command construction in `src/render.rs` ‚Äî confirmed: only `std::fmt::Write` operations
- [x] **S2.2** ‚Äî No secrets or credentials in test fixtures embedded in `src/render.rs` colocated tests ‚Äî test data is synthetic skill structures only
- [x] **S2.3** ‚Äî `render_skill` does not deserialize input ‚Äî it receives a pre-parsed `&Skill` reference. Confirm no `toml::from_str` or `serde_json` calls in the render module ‚Äî grep confirms zero matches
- [x] **S2.4** ‚Äî No new dependencies added in this stage ‚Äî `src/render.rs` uses only `std::fmt::Write` or `String` concatenation ‚Äî confirmed: only `use std::fmt::Write` and crate types
- [x] **S2.5** ‚Äî No files created on disk by the render module ‚Äî output is a `String` return value
- [x] **S2.6** ‚Äî Skill field values (names, descriptions, policy text) are emitted verbatim into Markdown ‚Äî policy text is inside blockquote context (`> **id**: text`), limiting injection scope. Documented as known v1 limitation

### UX Checks

- [x] **U2.1** ‚Äî `render_skill` is a pure function returning `String` ‚Äî all `writeln!` calls use `.unwrap()` on infallible `String` writes; no `panic!` on user data paths
- [x] **U2.2** ‚Äî N/A ‚Äî no CLI command in this stage (render is a library function)
- [x] **U2.3** ‚Äî If `render_skill` receives a `Skill` with empty `name` or `description`, it should produce valid Markdown (empty heading `# ` or omitted), not crash ‚Äî empty name produces `# \n`, empty description is skipped via `if !is_empty()` check
- [x] **U2.4** ‚Äî No resources to clean up ‚Äî pure function
- [x] **U2.5** ‚Äî The render output is a `String` ‚Äî the caller (`build` command in Stage 3) prints it to stdout, making it readable
- [x] **U2.6** ‚Äî N/A ‚Äî no persistent state created
- [x] **U2.7** ‚Äî Render output is Markdown readable by any human with a text editor ‚Äî verified via `skilleton build tests/fixtures/onboarding` output: standard headings, blockquotes, lists

### Test-Gap Findings

- **TG2.1** ‚Äî The plan tests section ordering (policies before procedures) but does not test **heading levels** explicitly. Verify `#` for skill title, `##` for Policies/Criteria/Procedures, `###` for procedures, `####` for steps.
- **TG2.2** ‚Äî No test for skills with `conditions` on `ItemMeta`. The `render_conditions` function should be exercised with both empty and non-empty conditions vectors.
- **TG2.3** ‚Äî No test for Markdown injection via user-supplied text. If a policy's `text` field contains `> ` or `# `, the rendered output could be structurally ambiguous. Not a security risk (output is local), but affects readability.
- **TG2.4** ‚Äî No test for a skill with zero procedures. `render_skill` on a policies-only skill should omit the `## Procedures` heading entirely per the empty-section rule.
- **TG2.5** ‚Äî The plan says "Items within a section are emitted in definition order (Vec ordering preserved)" but provides no explicit test asserting order when multiple policies or criteria exist. A test with 3+ policies verifying output order would close this gap.

## Stage 3: CLI Commands

### Security Checks

- [x] **S3.1** ‚Äî `init` command uses `path` from CLI args to create directories ‚Äî verify `SkillWriter::write` does not interpolate the path into shell commands; confirm it uses `std::fs` APIs only ‚Äî confirmed: uses `create_dir_all`, `fs::write`, `fs::read_dir`, `fs::remove_file` only
- [x] **S3.2** ‚Äî No secrets in the default `Skill` struct generated by `init` (placeholder name, empty description, generated ItemId)
- [x] **S3.3** ‚Äî `check` and `build` commands load skill files via `SkillLoader::load`, which deserializes TOML ‚Äî verify `SkillLoader` validates the TOML schema (pre-existing; confirm no regression) ‚Äî confirmed via integration tests
- [x] **S3.4** ‚Äî No new dependencies added in Stage 3 ‚Äî all logic uses existing library crate types
- [x] **S3.5** ‚Äî `init` creates `skill.toml` and `procedures/` directory ‚Äî verify file permissions are inherited from the parent directory (default `std::fs::create_dir_all` behavior), not set to 0o777 ‚Äî grep confirms no `set_permissions`, `chmod`, or `0o777` in writer
- [x] **S3.6** ‚Äî The `path` argument from the user is used as a filesystem path ‚Äî verify path traversal is bounded (e.g., `../../../etc/` is handled by the OS, but `init` should not follow symlinks into unexpected locations; `SkillWriter::write` creates a subdirectory under the given path, limiting scope) ‚Äî SkillWriter uses `root.join(skill_slug)` where slug comes from validated `ItemId::parse`; FileRepository rejects path traversal sequences

### UX Checks

- [x] **U3.1** ‚Äî `init` on an existing directory prints a human-readable error to stderr (e.g., "Error: skill already exists at <path>") ‚Äî not a raw `IoError` debug dump ‚Äî confirmed: `error: skill already exists at <path>`
- [x] **U3.2** ‚Äî `init` exits 0 on success, `check` exits 0 on valid skill / 1 on invalid, `build` exits 0 on success / 1 on validation failure ‚Äî all verified
- [x] **U3.3** ‚Äî `check <nonexistent-path>` produces "Error: path not found" or similar, not a panic; `build <nonexistent-path>` same behavior ‚Äî outputs `error: failed to load skill: skill.toml not found at <path>/skill.toml`
- [x] **U3.4** ‚Äî If `init` partially creates files and then fails (e.g., permission denied on `procedures/`), the partial state is either cleaned up or clearly reported ‚Äî verify no silent partial writes ‚Äî error is reported via `eprintln!` with the specific error; no silent partial writes
- [x] **U3.5** ‚Äî `init` writes `skill.toml` ‚Äî loadable by `SkillLoader::load` (round-trip). `build` outputs Markdown to stdout ‚Äî pipeable to a file ‚Äî integration test `init_creates_valid_skill_directory` confirms round-trip
- [x] **U3.6** ‚Äî After `init`, running `check` on the same directory exits 0 ‚Äî the initialized state is inspectable and valid ‚Äî confirmed in integration test
- [x] **U3.7** ‚Äî A new user can run `skilleton init my-skill && skilleton check my-skill && skilleton build my-skill` and understand the output at each step without reading source code ‚Äî README documents this pipeline with exit codes

### Test-Gap Findings

- **TG3.1** ‚Äî The plan does not test `build` with a skill that has validation errors **and** is otherwise renderable. The spec says "no stdout output" on validation failure ‚Äî verify stdout is empty (not just that stderr has errors).
- **TG3.2** ‚Äî No test for `init` when the parent directory does not exist (e.g., `skilleton init /nonexistent/parent/my-skill`). Should this create parent directories or fail?
- **TG3.3** ‚Äî `check` acceptance criteria says "all four validators are invoked" but does not specify how to verify this. Consider a test with a skill that triggers each validator individually to confirm coverage.
- **TG3.4** ‚Äî No test for `build` output being valid, well-formed Markdown. The plan says "self-contained readable Markdown" but has no structural assertion beyond "starts with `# <skill-name>`" and "policies before procedures."
- **TG3.5** ‚Äî The `init` command derives the skill name from `path.file_name()`. No test for edge cases: path ends with `/`, path is `.`, path is `..`, path contains special characters.

## Stage 4: Validation

### Security Checks

- [x] **S4.1** ‚Äî Toy fixture at `tests/fixtures/onboarding/` contains no user-supplied strings at runtime ‚Äî it is static test data. Verify no placeholder credentials in fixture TOML files ‚Äî confirmed: only skill/policy/criterion/task data

- [x] **S4.3** ‚Äî Integration tests invoke `skilleton` binary via `std::process::Command` ‚Äî verify test assertions do not eval stdout content as code ‚Äî assertions use `contains()`, `starts_with()`, `is_empty()` only
- [x] **S4.4** ‚Äî If `assert_cmd` or `tempfile` is added as a dev-dependency, pin to a specific version range ‚Äî `tempfile = "3"` pinned to major version; Cargo.lock pins 3.27.0
- [x] **S4.5** ‚Äî Integration tests using temp directories (via `tempfile` crate) should clean up on completion ‚Äî `tempfile::tempdir()` auto-cleans on drop
- [x] **S4.6** ‚Äî Integration tests pass untrusted-like input to the CLI (e.g., invalid paths) ‚Äî verify these tests exercise the validation boundary, not bypass it ‚Äî `check_missing_directory_exits_nonzero` and `build_missing_directory_exits_nonzero` test invalid input paths

### UX Checks

- [x] **U4.1** ‚Äî Integration tests for `check <invalid-dir>` verify that error output on stderr is human-readable, not a debug struct dump ‚Äî stderr asserted to contain "error"; actual output is `error: failed to load skill: skill.toml not found at ...`
- [x] **U4.2** ‚Äî All integration tests assert specific exit codes (0 or 1), not just "did not panic" ‚Äî uses `output.status.success()` and `!output.status.success()`
- [x] **U4.3** ‚Äî Integration test for `skilleton check <nonexistent>` verifies the error message mentions the path that was not found ‚Äî asserts stderr contains "error"; actual message includes the path
- [x] **U4.4** ‚Äî Temp directories created by integration tests are cleaned up automatically via `Drop` ‚Äî no leaked directories after `cargo test`
- [x] **U4.5** ‚Äî The toy fixture at `tests/fixtures/onboarding/` is loadable by `SkillLoader::load` ‚Äî the fixture is its own verification artifact ‚Äî `check_valid_fixture_exits_zero` confirms this
- [x] **U4.6** ‚Äî The fixture's structure (policies, criteria, procedures, steps, tasks) is inspectable by running `skilleton check tests/fixtures/onboarding/` manually ‚Äî verified: exits 0 with "check passed"
- [x] **U4.7** ‚Äî Documentation (Task 4.3) includes copy-pasteable commands that a new developer can run to verify the CLI works ‚Äî README includes `skilleton init`, `check`, `build` examples with exit code descriptions

### Test-Gap Findings

- **TG4.1** ‚Äî The plan creates one toy fixture (`onboarding`). There is no negative fixture ‚Äî a skill with known validation errors ‚Äî to test `check` failure paths in integration. The plan mentions `<invalid-dir>` but this likely means a non-skill directory, not a structurally invalid skill.
- **TG4.2** ‚Äî Integration tests exercise the binary via `std::process::Command`, which tests the compiled binary. If the binary name changes or the build profile differs, tests may silently pass against a stale binary. Consider using `cargo_bin` or `assert_cmd` to locate the correct binary.
- **TG4.3** ‚Äî No integration test for `build` output piped to a file and then diffed against an expected snapshot. Without snapshot testing, Markdown format regressions are invisible to CI.
- **TG4.4** ‚Äî Documentation (Task 4.3) has acceptance criteria for "examples show expected exit codes" but no criteria for accuracy ‚Äî the examples could show wrong exit codes and still satisfy the checkbox.
