## Stage 1: Serialization Foundation

### Security Checks

- [ ] **S1.1** — `toml` dependency version `"0.8"` is a SemVer range, not a pinned version. Verify that `Cargo.lock` locks to a specific version and that the range does not admit a future 0.8.x with breaking behavior. `serde = "1"` is similarly a range.
- [ ] **S1.2** — `ItemId::deserialize` calls `ItemId::parse`, which validates type prefixes and slug charset. Verify that no path through deserialization bypasses `ItemId::parse` validation — this is the boundary validation for untrusted TOML input.
- [ ] **S1.3** — `CriterionRef::deserialize` accepts any valid `ItemId` regardless of type prefix. A TOML file containing `completion_criteria = ["skill:foo"]` would deserialize without error. Verify whether type-prefix validation belongs at the deserialization boundary or is deferred to semantic validation.
- [ ] **S1.4** — `#[serde(flatten)]` on `ItemMeta` inlines `id` and `conditions` into parent TOML tables. When combined with TOML arrays-of-tables for sibling fields (`policies`, `steps`), verify that the `toml` crate does not silently drop or misroute fields. This is a known fragility point — the `toml` crate internally uses `serde::de::Content` for flattened deserialization, which may not handle all TOML value types correctly.
- [ ] **S1.5** — Two `#[serde(flatten)]` attributes in `SkillManifest` (both `ItemMeta` and `SkillMeta` are flattened). Verify no field-name collisions or silent key conflicts between the two flattened structs. Current fields (`id`, `conditions` vs `name`, `description`) do not collide, but this is a fragile pattern.
- [ ] **S1.6** — No secrets or credentials appear in test fixture TOML strings.

### UX Checks

- [ ] **U1.1** — Malformed TOML deserialization produces a clear `serde::de::Error` message, not a panic. Verify with: missing required fields, wrong value types, invalid ItemId strings.
- [ ] **U1.2** — The `ItemId` serde error message from `ItemId::parse` failures is human-readable via `serde::de::Error::custom`. Verify the error message includes the invalid string and the reason for rejection.
- [ ] **U1.3** — TOML serialization output for a Skill is readable and round-trips. A user inspecting the TOML output can understand the structure without consulting code.
- [ ] **U1.4** — Optional fields (`invokes`, `compatible_with`) are cleanly absent from TOML output when `None`/empty, not present as empty strings or null sentinels.

### Test-Gap Findings

**TG1.1** — `CriterionRef` deserialization accepts any valid `ItemId` type prefix. A criterion reference containing `"skill:foo"` or `"task:bar"` deserializes without error. The dev tests verify `CriterionRef` round-trips through its inner `ItemId` but do not test rejection of non-`criterion:*` prefixed IDs. If this validation is intentionally deferred, the gap should be documented. If not, it is a silent data integrity issue.

**TG1.2** — `#[serde(flatten)]` with the `toml` crate is fragile for structs containing `Vec` fields alongside flattened fields. The plan acknowledges this ("fall back to DTO types") but no acceptance criterion covers the fallback path. If flatten works, the fallback code is never written. If flatten fails, the implementer discovers it at test time with no pre-defined acceptance criteria for the DTO approach. Either path needs explicit coverage.

**TG1.3** — New `criteria: Vec<Criterion>` fields on `Skill`, `Procedure`, and `Step` require `#[serde(default)]` for backward-compatible deserialization. The plan shows `#[serde(default, skip_serializing_if = "Vec::is_empty")]` for `compatible_with` but does not explicitly state `#[serde(default)]` for the `criteria` fields. Without it, TOML files omitting `criteria` will fail to deserialize.

**TG1.4** — No acceptance criterion tests TOML containing unknown/extra fields. The default serde behavior with the `toml` crate is to reject unknown fields unless `#[serde(deny_unknown_fields)]` is absent. The plan should specify whether extra fields are silently ignored (forward compatibility) or rejected (strict parsing). With `#[serde(flatten)]`, unknown field handling is further complicated — flattened deserializers may consume unexpected keys.

**TG1.5** — No acceptance criterion tests TOML strings containing characters that require escaping (quotes, backslashes, newlines in policy text). The `toml` crate handles this, but a round-trip test with `text = "She said \"hello\""` or multi-line strings would verify no data corruption.

**TG1.6** — Existing tests construct `Policy` with two fields (`meta`, `text`). Task 1.3 adds `compatible_with`. The plan says "update existing tests" but does not specify what "update" means. If tests are updated to add `compatible_with: vec![]`, they compile but do not exercise the new field. Verify that at least one existing test is updated to use a non-empty `compatible_with`.

## Stage 2: Policy Conflict Detection

### Security Checks

- [ ] **S2.1** — `compute_effective_policies` operates on in-memory `&Skill` data. No external input surfaces. Verify the function does not construct file paths, shell commands, or SQL from policy text or ItemId strings.
- [ ] **S2.2** — `compatible_with: Vec<ItemId>` values are parsed via `ItemId::deserialize`, which validates format. Verify no code path uses `compatible_with` IDs to construct file paths or perform lookups without checking existence first.
- [ ] **S2.3** — Policy text is carried through the overlap report as-is. Verify that downstream consumers of `PolicyOverlap` do not interpolate policy text into format strings, log messages, or error outputs without escaping.

### UX Checks

- [ ] **U2.1** — The `PolicyOverlap` struct is structured data (not free-form text). Verify that the `Debug` or `Display` implementation on `PolicyOverlap` produces a human-readable summary including: target scope, policy IDs, origin scopes, and overlap type.
- [ ] **U2.2** — `detect_policy_overlaps` returns an empty `Vec` (not an error) when no overlaps exist. A clean skill produces clean output.
- [ ] **U2.3** — When `compatible_with` annotations suppress an overlap, the suppressed overlap is absent from output. There is no "suppressed" status reported — suppression is silent. Verify this matches the user's expectation (can a user tell that annotations are working?).

### Test-Gap Findings

**TG2.1** — `compute_effective_policies` must handle `Task.invokes` references to callee Procedures. If the referenced procedure does not exist in the Skill (a dangling reference), the function's behavior is unspecified. The `validate.rs` module catches this, but `compute_effective_policies` may be called independently. No acceptance criterion covers this error path.

**TG2.2** — Overlap detection for three or more converging policies. The compatibility check (`are_compatible`) requires mutual annotations between ALL pairs. For three policies A, B, C, suppression requires A↔B, A↔C, and B↔C (three mutual annotations). No acceptance criterion tests the N>2 case, where some pairs are annotated but not all. A partially-annotated three-way overlap could silently remain in the report or be silently suppressed depending on implementation.

**TG2.3** — `compatible_with` references are not validated against existing policy IDs. A policy with `compatible_with = ["policy:typo-nonexistent"]` compiles and deserializes without error but silently fails to suppress any overlap. No acceptance criterion verifies that dangling `compatible_with` references produce a warning or error.

**TG2.4** — Self-referential `compatible_with` annotation: a policy listing its own ID in `compatible_with`. This is logically meaningless and may indicate a user error. No acceptance criterion specifies whether this is ignored, warned, or rejected.

**TG2.5** — Multiple Tasks in a single Step invoking different Procedures. Each invocation creates its own cross-procedure merge point. The plan's tests show one invocation per step. A Step with two tasks invoking different Procedures creates two independent overlap contexts at the same parent scope — verify the overlap report distinguishes them.

**TG2.6** — Diamond invocation pattern: Procedure A invokes B and C; both B and C define policies. At A's scope, B's and C's policies converge through independent invocation paths. This is distinct from a single cross-procedure merge and exercises the overlap detector's ability to handle transitive convergence.

## Stage 3: File Organization

### Security Checks

- [ ] **S3.1** — Slug-to-filename mapping derives from `ItemId` slugs. `ItemId::parse` restricts slugs to `[a-z0-9-]` with max 50 chars — no `..`, `/`, `\`, or null bytes. Verify that `SkillWriter` uses only validated slug values for directory/file names and does not construct paths from unvalidated strings.
- [ ] **S3.2** — `SkillWriter::write` creates directories and files. Verify created files have default permissions (not world-writable). On Unix, `std::fs::create_dir_all` and `std::fs::write` use the process umask — verify this is acceptable.
- [ ] **S3.3** — `SkillLoader::load` reads from a user-supplied `Path`. Verify it does not follow symlinks outside the skill directory (symlink traversal). A `procedures/malicious.toml` symlink pointing to `/etc/passwd` would be read and parsed (and fail to parse, but the file contents would be read into memory).
- [ ] **S3.4** — `SkillLoader` reads all `*.toml` files in the `procedures/` directory. An attacker placing a crafted `.toml` file in the directory could inject arbitrary TOML data. The deserialization boundary validation (serde + ItemId::parse) is the defense. Verify there is no code path that bypasses it.
- [ ] **S3.5** — `SkillWriter::write` behavior when the target directory already exists is unspecified. If it overwrites existing files without warning, a user could lose data. If it fails, the error should be clear.
- [ ] **S3.6** — No secrets in test fixture data or generated TOML files.

### UX Checks

- [ ] **U3.1** — `StorageError::MissingSkillFile` includes the expected file path so the user knows where to create it.
- [ ] **U3.2** — `StorageError::ParseError` includes both the file path and the parse error detail, so the user can find and fix the malformed TOML.
- [ ] **U3.3** — `StorageError::IoError` includes the file path and the OS-level error, not just `std::io::Error` without context.
- [ ] **U3.4** — `StorageError` implements `Display` and `std::error::Error`. Without `Display`, error messages default to `Debug` format, which is not human-readable. The plan defines the enum variants but does not show trait implementations.
- [ ] **U3.5** — A user can verify `SkillWriter` output by inspecting the generated directory tree and reading individual TOML files. The TOML files should be self-contained and readable without the tool.
- [ ] **U3.6** — `SkillLoader::load` returns a meaningful error when `procedures/` directory is missing entirely (not empty). The plan tests empty `procedures/` but not absent `procedures/`.

### Test-Gap Findings

**TG3.1** — `SkillWriter::write` overwrite behavior is undefined. If the target skill directory already exists (e.g., from a previous write), the implementation could: (a) overwrite all files silently, (b) return an error, or (c) merge. No acceptance criterion specifies the behavior. Silent overwrite risks data loss if a user has manually edited procedure files. An error prevents iterative write workflows.

**TG3.2** — `SkillLoader` reads `procedures/*.toml` — but what about non-`.toml` files in the directory? A `README.md`, `.gitkeep`, or `.DS_Store` file in `procedures/` could cause unexpected behavior. No acceptance criterion specifies whether non-TOML files are ignored or produce errors.

**TG3.3** — Procedure ordering after load is filesystem-dependent. `std::fs::read_dir` returns entries in OS-defined order (not alphabetical on all platforms). If the order of `Skill.procedures` matters for display, validation, or diff stability, the loader must sort. No acceptance criterion specifies ordering, so `SkillWriter::write → SkillLoader::load` round-trip may fail equality checks if the procedures Vec order differs.

**TG3.4** — `StorageError` enum is defined but the plan does not show `Display` or `std::error::Error` implementations. Without them, the UX checklist items (human-readable error messages) cannot be satisfied. This is a code-level omission that the acceptance criteria do not catch.

**TG3.5** — Round-trip integration test (Task 3.3) uses `assert_eq!(skill, loaded)`. This requires `PartialEq` on `Skill` and all contained types — which already exists. But if procedure ordering (TG3.3) differs, the assertion fails non-deterministically. The test may pass on one OS and fail on another.

**TG3.6** — `SkillLoader` does not validate that the loaded Procedures' ItemId slugs match their filenames. A file named `auth-flow.toml` containing `id = "procedure:welcome"` would load without error. The slug-filename mismatch is a silent data integrity issue.

## Stage 4: Item Storage and Retrieval

### Security Checks

- [ ] **S4.1** — `FileRepository::load_skill` constructs a path by joining `self.root` with the user-supplied `name` parameter: `self.root.join(name)`. If `name` contains `..` or `/`, this enables directory traversal outside the root. Verify that `name` is validated as a slug (or that `SkillLoader::load` rejects paths outside the expected structure).
- [ ] **S4.2** — `FileRepository::list_skills` scans the root directory. Verify it does not follow symlinks to directories outside the root, and does not return directory names that would enable traversal when passed back to `load_skill`.
- [ ] **S4.3** — `InMemoryRepository` stores skills in a `HashMap<String, Skill>`. The key is the skill name (a string). Verify the name is not used to construct file paths or shell commands downstream.
- [ ] **S4.4** — `RepositoryError` enum wraps `StorageError`. Verify error Display implementations do not leak internal file system paths to untrusted callers in contexts where that would be a concern.

### UX Checks

- [ ] **U4.1** — `RepositoryError::NotFound` message includes the skill name the user requested, so they can correct typos.
- [ ] **U4.2** — `RepositoryError` implements `Display` and `std::error::Error`. The plan defines the enum but does not show trait implementations. Without them, errors are printed in `Debug` format.
- [ ] **U4.3** — `find_item` returns `Ok(None)` for a nonexistent ItemId, not an error. Verify this distinction is documented — `NotFound` is for missing skills, `None` is for missing items within a skill.
- [ ] **U4.4** — `list_skills` returns an empty `Vec` (not an error) when the root directory exists but contains no skills. Verify this matches the user's expectation.
- [ ] **U4.5** — A user can verify repository state by calling `list_skills` and then `load_skill` for each. The API supports state inspection.

### Test-Gap Findings

**TG4.1** — `FileRepository::load_skill` passes user-supplied `name` directly to `self.root.join(name)`. No validation that `name` is a valid slug. An input of `"../../../etc"` would attempt to load from outside the root. `SkillLoader::load` would fail with `MissingSkillFile` (no `skill.toml` at that path), but the path traversal attempt is itself a security concern that should be rejected at the `load_skill` boundary.

**TG4.2** — `find_item` walks the Skill hierarchy matching by `ItemId`. If two items at different scopes have the same `ItemId` (e.g., `policy:no-plaintext` at both Skill and Procedure level), `find_item` returns the first match in traversal order. No acceptance criterion specifies which match wins or whether duplicates produce an error.

**TG4.3** — `FileRepository::find_item` calls `self.load_skill(skill)?` which reads from disk on every invocation. For repeated `find_item` calls on the same skill, this means repeated disk I/O and deserialization. No acceptance criterion tests performance characteristics, but the behavior should be documented.

**TG4.4** — `FileRepository::list_skills` scans the root for subdirectories containing `skill.toml`. If the root directory does not exist, the behavior is unspecified — it could return an empty vec, return an I/O error, or panic. No acceptance criterion covers a non-existent root.

**TG4.5** — `InMemoryRepository::add_skill` is called with a name and a Skill. If the name differs from the Skill's ItemId slug (e.g., `add_skill("foo", skill_with_id_bar)`), the repository stores the skill under a mismatched name. No acceptance criterion verifies name-ID consistency.

**TG4.6** — `RepositoryError` and `StorageError` need `Display` and `std::error::Error` implementations. The plan defines the enum variants but omits trait impls from both the Implementation Notes and Acceptance Criteria. This means all error paths produce `Debug` format output, violating the UX checklist (human-readable messages on stderr).

## Stage 5: Finalize

### Security Checks

- [ ] **S5.1** — ADR status update is a text edit, no new input surfaces. Verify only the `Status` line is changed and no other ADR content is modified.

### UX Checks

- [ ] **U5.1** — The status transition is from `Planned` to `Accepted`. The plan notes that ADRs will be in `Planned` status at execution time. Verify the actual status string in the file before editing — if the ADR is still `Ready` (not yet transitioned to `Planned`), the edit would create an incorrect status progression.

### Test-Gap Findings

**TG5.1** — The plan says ADRs will be in `Planned` status, but the current ADR files show `Status: Ready`. The implementation plan's Step I-5 (referenced in the note) transitions them to `Planned` before execution. If execution starts without that transition, Task 5.1's acceptance criteria ("each ADR status reads Accepted") would be met, but the audit trail would show Ready → Accepted, skipping `Planned`. No acceptance criterion verifies the pre-condition status.

## Cross-Cutting Findings

**TG-X1** — `#[serde(flatten)]` is a systemic risk across Stages 1 and 3. The `toml` crate's flatten support has known limitations with complex types (arrays of tables, nested structures). The plan uses flatten in: `ItemMeta` on every hierarchy type (Task 1.3), `SkillMeta` on `SkillManifest` (Task 3.1), and both `ItemMeta` and `SkillMeta` on `SkillManifest` simultaneously (double-flatten). If any of these fail, the fallback (DTO types) requires redesigning the serialization layer mid-implementation. No acceptance criterion in any stage tests the flatten-specific behavior in isolation.

**TG-X2** — Error type `Display` implementations are missing from the plan's acceptance criteria across Stages 3 and 4. `StorageError` (Task 3.2) and `RepositoryError` (Task 4.1) are defined as enums but no task requires `impl Display` or `impl std::error::Error`. Every UX check requiring "human-readable error messages" depends on these impls existing.

**TG-X3** — Procedure ordering across write-then-load is non-deterministic. `SkillWriter` writes procedures to individual files (Task 3.1), `SkillLoader` reads them via `fs::read_dir` (Task 3.2), which returns OS-dependent order. The round-trip integration test (Task 3.3) uses `assert_eq!`, which compares Vec ordering. This test may pass on Linux (which often returns inode order ≈ creation order) and fail on other platforms. Sorting procedures by slug after loading would fix this, but neither the implementation notes nor acceptance criteria mention ordering.
