## Stage 1: Foundation Types

Stage 1 defines pure data structs with no I/O, no CLI surface, no user input, and no dependencies beyond `std`. Most security and UX checks are not applicable.

### Security Checks

| ID | Check | Applicable | Notes |
|----|-------|------------|-------|
| S1.1 | No user-input injection | N/A | No user input; types are constructed in-code |
| S1.2 | No secrets in source | Yes | Verify no API keys, tokens, or credentials in committed code |
| S1.3 | No untrusted deserialization | N/A | No serde derives or parsing in Stage 1 |
| S1.4 | Pinned dependencies | N/A | No new dependencies added |
| S1.5 | File permissions | N/A | No file I/O; only source files created |
| S1.6 | Input validation at boundaries | N/A | No public API boundary yet; all fields are `pub` by design |

### UX Checks

| ID | Check | Applicable | Notes |
|----|-------|------------|-------|
| U1.1 | Error messages on stderr | N/A | Library crate, no CLI output |
| U1.2 | Exit codes | N/A | Library crate, no binary target |
| U1.3 | Invalid input rejection | N/A | No constructors or validation logic in Stage 1 |
| U1.4 | Resource cleanup | N/A | No resources allocated; all types are plain data |
| U1.5 | Data read-back | N/A | No persistence layer |
| U1.6 | State inspection | Partial | `Debug` derive on all types enables `dbg!()` / `{:?}` formatting |
| U1.7 | Output verification | N/A | No output beyond compilation |

### Test-Gap Findings

| ID | Finding |
|----|---------|
| TG1.1 | **No `PartialEq` / `Eq` on `ItemMeta`, `Policy`, `Criterion`, hierarchy types.** Plan derives `Debug` and `Clone` but not `PartialEq`. Stage 2 tests will need manual field-by-field comparison or future derive additions. |
| TG1.2 | **`SkillMeta` derives `Default` but other types do not.** If any test constructs a "minimal" `ItemMeta` or `Step`, it must populate all fields manually. Inconsistent `Default` coverage could mask missing-field bugs in tests. |
| TG1.3 | **Public field invariants are untested.** All fields are `pub`, so nothing prevents constructing a `Procedure` with mismatched `entrance_criteria` referencing nonexistent `CriterionRef`s. The plan doesn't address this (by design for 0.x), but it should be explicitly noted as a deferred concern. |
| TG1.4 | **`Hash` not derived on `ItemId`.** `ItemId` derives `Eq` but not `Hash`, so it cannot be used as a `HashMap` key. If downstream work needs ID lookups, this will surface as a compile error rather than a test failure. |

### Recommendations

| # | Stage | Classification | Finding | Recommendation | Plan Ref | Status |
|---|-------|---------------|---------|----------------|----------|--------|
| 1 | 1 | Quality concern | `PartialEq` missing on most types | Derive `PartialEq` on `ItemMeta`, `Policy`, `Criterion`, and hierarchy types so Stage 2 tests can use `assert_eq!` | Task 1.2, 1.3 | Open |
| 2 | 1 | Preference | `Hash` missing on `ItemId` | Derive `Hash` on `ItemId` and `CriterionRef` for future use in collections | Task 1.1 | Open |
| 3 | 1 | Quality concern | `SkillMeta` has `Default` but `ItemMeta` does not | Either add `Default` to `ItemMeta` (with a placeholder `ItemId`) or remove `Default` from `SkillMeta` to keep derive consistency | Task 1.1, 1.2 | Open |
| 4 | 1 | Preference | No doc comments on public types | Add `///` doc comments to each public type and field; `#[warn(missing_docs)]` would enforce this | Task 1.1–1.3 | Open |

## Stage 2: Unit Tests

Stage 2 writes unit tests for the types defined in Stage 1. Security checks focus on test hygiene. UX checks focus on test ergonomics and diagnostic quality.

### Security Checks

| ID | Check | Applicable | Notes |
|----|-------|------------|-------|
| S2.1 | No user-input injection | N/A | Tests use hardcoded literals |
| S2.2 | No secrets in source | Yes | Verify test data contains no real credentials or PII |
| S2.3 | No untrusted deserialization | N/A | No deserialization in test scope |
| S2.4 | Pinned dependencies | N/A | No test-only dependencies added |
| S2.5 | File permissions | N/A | No file I/O in tests |
| S2.6 | Input validation at boundaries | N/A | Tests exercise internal construction, not public API boundaries |

### UX Checks

| ID | Check | Applicable | Notes |
|----|-------|------------|-------|
| U2.1 | Error messages on stderr | N/A | Library tests, no stderr output |
| U2.2 | Exit codes | Partial | `cargo test` exit code must be 0; verified by acceptance criteria |
| U2.3 | Invalid input rejection | N/A | No validation logic to test |
| U2.4 | Resource cleanup | N/A | No resources to clean up |
| U2.5 | Data read-back | Yes | Tests should verify that values stored in struct fields are readable back via field access |
| U2.6 | State inspection | Yes | Tests should use `Debug` formatting to provide useful failure messages |
| U2.7 | Output verification | Yes | `assert_eq!` or equivalent on constructed values |

### Test-Gap Findings

| ID | Finding |
|----|---------|
| TG2.1 | **No `Clone` round-trip test.** All types derive `Clone`, but no test verifies that cloning produces an equivalent value. Without `PartialEq`, clone correctness is unverifiable. |
| TG2.2 | **No negative composition test.** Plan criterion 1.3 says "a Step cannot contain a Procedure" and marks it as compile-time enforced, but no test (even a compile-fail test) demonstrates this. Consider a doc comment or a `trybuild` test. |
| TG2.3 | **Empty hierarchy tests missing.** No test constructs a `Skill` with zero procedures, or a `Procedure` with zero steps. These are valid states (empty `Vec`) and should be exercised. |
| TG2.4 | **`CriterionRef` referential integrity untested.** A `CriterionRef` can point to an `ItemId` that doesn't correspond to any `Criterion`. No test verifies this is acceptable (or documents it as a known gap). |
| TG2.5 | **Multi-policy, multi-criteria tests missing.** Task 2.2 tests attachment but doesn't specify testing with multiple policies at the same level or criteria lists with more than one entry. |

### Recommendations

| # | Stage | Classification | Finding | Recommendation | Plan Ref | Status |
|---|-------|---------------|---------|----------------|----------|--------|
| 5 | 2 | Quality concern | No empty-hierarchy test | Add test: `Skill` with empty `procedures`, `Procedure` with empty `steps`, `Step` with empty `tasks` | Task 2.1 | Open |
| 6 | 2 | Quality concern | Clone correctness unverifiable | If `PartialEq` is added (Rec #1), add a clone round-trip `assert_eq!` test for at least one hierarchy type | Task 2.1 | Open |
| 7 | 2 | Preference | No compile-fail test for invalid composition | Add a `trybuild` compile-fail test or a doc comment demonstrating that `Step { tasks: vec![some_procedure] }` fails to compile | Task 2.1 | Open |
| 8 | 2 | Quality concern | Multi-element attachment not tested | Add test with ≥2 policies on one `Step` and ≥2 `CriterionRef`s in `entrance_criteria` | Task 2.2 | Open |
| 9 | 2 | Preference | `CriterionRef` dangling reference undocumented | Add a test or doc comment acknowledging that referential integrity is the caller's responsibility | Task 2.1 | Open |
